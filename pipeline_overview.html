Phase 1 Base Dataset — OpenStreetMap Done
Why this phase? Before any modelling can happen, you need a reliable, reproducible list of venues with ground-truth locations. OSM is the most comprehensive openly licensed geospatial database in the world. Using it as a base means your dataset is free, reproducible, and not locked behind a commercial API. Every subsequent step builds on this foundation.
✓
Overpass API query
Query amenity=cafe, coffee_shop, shop=coffee within Copenhagen bounding box. Nodes + ways with center coordinates.
OSM stores cafés under multiple tag conventions — using only amenity=cafe would miss venues tagged differently by different contributors. Querying all three tag types maximises recall.
osm_cafe_fetch.py → data/cafes_raw.json
✓
Map visualisation
Interactive HTML map with bounding box rectangle and a pin per venue.
Visually verifies the bounding box is correct and the returned venues are geographically sensible before investing time in downstream processing. Also useful for thesis figures.
plot_cafes_on_map.py → data/cafes_map.html
○
Tag frequency analysis
Count all unique OSM tags across all 1,009 venues. Calculate fill rate (% of venues that have each tag). Save as a ranked table.
Feature selection must be justified methodologically. You cannot just pick tags arbitrarily — you need to show which tags exist, how complete they are, and argue why you kept or discarded each one. This table is that justification.
○
Spatial join — Copenhagen bydele
Fetch the 10 official borough polygons from Copenhagen municipality's open data WFS API. Run a point-in-polygon test to assign each café a district label.
Raw lat/lon coordinates are not interpretable on their own. District labels provide a meaningful spatial variable that is human-readable, statistically tractable, and directly relevant to the thesis question of spatial distribution. They also serve as a categorical feature in downstream models.
outputs → cafes_raw.json cafes_map.html cafes_with_district.csv tag_frequency.csv

Phase 2 Data Enrichment — Filling the Gaps Next
Why this phase? OSM data alone is too sparse to build meaningful representations. Most venues have a name and coordinates, but little else. The richest signal for understanding what a café is — its menu, its atmosphere, its offering — lives on the open web. This phase transforms a thin structured dataset into a rich one by pulling in unstructured web content. It also addresses a core thesis challenge: noisy and incomplete data.
⚑
The core challenge: ~531 venues have no website in OSM
These venues are not necessarily offline — OSM contributors simply haven't added their website. The supervisor's guidance: don't discard them, find their data elsewhere.
Discarding half your dataset based on a missing field would introduce systematic bias — smaller or less digitally active venues would be underrepresented. Attempting to fill the gap, and documenting what you could and couldn't find, is itself a contribution to the literature on incomplete observational data.
Strategy A — Venues WITH a website (475)
Fetch homepage, score all links for menu relevance
Follow highest-scoring links up to 3 levels deep
Detect and extract PDF menus (pdfplumber)
Fallback to Playwright for JS-rendered sites
Save: menu text, prices, descriptions, atmosphere
The website is the most authoritative source for a venue's own description of itself — menu items, pricing, and tone all signal what kind of place it is. This is your richest text source for embeddings.
Strategy B — Venues WITHOUT a website (531)
Search for "<name> Copenhagen" via free search
Check Google Maps / TripAdvisor / Yelp listing
Extract: reviews, category, description, price level
Record source URL and flag as "third-party data"
Third-party review platforms often have richer data than OSM for popular venues — user reviews, categories, and price levels. Tracking the source is essential for transparency and reproducibility in the thesis.

Phase 3 Feature Engineering — Building Representations Future
Why this phase? Machine learning models cannot operate on raw text or mixed data directly. This phase converts everything collected so far into a numerical form that captures the semantic and structural properties of each café. This is the central methodological contribution of the thesis — the representation itself.
○
Text cleaning
Strip navigation, cookie banners, and boilerplate. Detect language (Danish vs English). Identify menu items using price pattern anchors (e.g. 145,-).
Raw scraped HTML contains large amounts of noise — header links, footer text, legal notices — that would pollute embeddings and obscure the signal. Cleaning ensures the model encodes the café's actual content, not its website template.
○
Text embeddings — open source multilingual model
Encode menu and description text using paraphrase-multilingual-MiniLM-L12-v2. Produces a dense vector per café capturing semantic meaning. Handles Danish and English natively.
Dense embeddings capture semantic similarity that keyword matching cannot. Two cafés with different menu item names but similar concepts (e.g. "avocado toast" and "avocado på surdej") will be placed close together in the embedding space. The multilingual model is necessary because Copenhagen café menus are a mix of Danish and English.
HuggingFace sentence-transformers — free, runs locally
○
Unified feature vector per café
Concatenate text embeddings with encoded structured attributes: cuisine (one-hot), district (one-hot), opening hours (binary flags), outdoor seating, wheelchair accessibility.
Text alone captures what a café serves, but not where it is or how it operates. Combining text embeddings with structured features produces a richer representation that reflects both the venue's identity and its operational context — necessary for the spatial analysis and knowledge completion tasks.
outputs → features.csv embeddings.npy

Phase 4 Unsupervised Learning — Clustering & Embeddings Future
Why this phase? With no predefined category labels, unsupervised learning lets the data reveal its own structure. The goal is to discover latent venue types that emerge from the representations — not categories imposed top-down. Examining whether these clusters align with spatial patterns tests whether the representations capture something real about urban geography.
○
Clustering — K-Means / HDBSCAN
Run clustering algorithms on the feature vectors. Evaluate cluster quality (silhouette score, Davies-Bouldin). Examine spatial distribution of clusters across bydele.
K-Means is simple and interpretable but assumes spherical clusters. HDBSCAN handles arbitrary shapes and naturally identifies outliers — useful since some venues will be hard to classify. Running both lets you compare and choose based on your data's structure.
○
Dimensionality reduction — UMAP / t-SNE
Project high-dimensional feature vectors to 2D. Colour points by cluster label, district, or cuisine type.
High-dimensional vectors cannot be directly visualised. UMAP preserves both local and global structure, making it better suited than t-SNE for understanding cluster separability. The resulting 2D map is a key thesis figure — it makes the learned representations tangible.
outputs → cluster_labels.csv umap_projection.csv

Phase 5 Knowledge Completion — Predicting Missing Attributes Future
Why this phase? A large fraction of venues are missing attributes like cuisine type, price level, or opening hours. Rather than treating this as a data quality problem to be discarded, this phase treats it as a research problem: can the learned representations be used to infer what is missing? This is the thesis's direct contribution to the literature on knowledge graph completion and inference under incomplete observational data.
○
Predictive models
Treat missing attributes (cuisine, price level, opening hours) as prediction targets. Train classifiers on venues with known values, evaluate on held-out set, then predict for venues where the attribute is missing.
This closes the loop between data collection and representation learning — the quality of predictions directly reflects the quality of the representations built in Phase 3. Good representations should make missing attributes predictable from what is known.
○
Evaluation & analysis
Measure accuracy, precision, recall per attribute. Analyse feature importance. Discuss which venue characteristics are most predictable and why.
Evaluation contextualises the results — a model that predicts cuisine at 80% accuracy on a 10-class problem is meaningful, but only if you show the baseline (random = 10%) and discuss failure cases. This is where the thesis argument is made.
outputs → predictions.csv model_evaluation.json
